# Known Issues

Відомі проблеми поточної реалізації з планом вирішення. Виправлення планових
речей з беклогу — в `docs/potential-improvements.md`; тут — те, що вже сьогодні
дає некоректні або оманливі результати.

---

## 1. List-only снепшоти проходять field-compare і дають хибний «Different»

### Проблема

Адаптери з **двоетапним збором** (список ID → окремий запит деталей на кожну
сутність) обмежують кількість detail-запитів, щоб не робити сотні послідовних
HTTP-викликів. Сутності понад кап потрапляють у результат **з неповним
снепшотом** — лише поля, доступні зі списку.

Наслідки:

1. **Повнота не страждає** — множина ID для перевірки «є в БД / немає в БД»
   береться зі списку, кап її не ріже (Ryde). ✓
2. **Field-compare шумить** — незбагачені сутності зараз *не пропускаються*,
   а порівнюються з неповними даними: `battery` dynamic (ігнорується), але
   `name = null` проти реального значення в БД — static-поле → хибний вердикт
   **«Different»** у AI Results / Manual Review.
3. Тонша асиметрія: для збагачених сутностей адаптер віддзеркалює фільтр
   скрапера («немає battery/GPS у деталях → рядок не потрапляє в БД» → теж
   викидаємо). Для сутностей понад кап фільтр не застосовується — теоретично
   можливий рідкісний хибний «missing in DB».

### Де це в коді

| Адаптер | Кап | Що ріже |
|---------|-----|---------|
| `ryde-adapter.ts` — `MAX_VEHICLE_DETAILS = 50` (на тайл) | деталі dockless | **лише збагачення**; ID-множина повна. Практично кап майже не спрацьовує: у Trondheim/Bærum ~10 флітів на тайл (кап — запобіжник від патологічного тайла) |
| `lyft-adapter.ts` — `MAX_PRICING_STATIONS = 5` (на полігон) | pricing-запити по станціях | **саму множину ID**: pricing-и збираються тільки з 5 станцій → повнота pricings для Lyft оцінюється частково (сутності інших станцій взагалі не потрапляють в API-множину). Інший прояв тієї самої проблеми — «мовчазна» неповнота без жодної позначки в результатах |

Порівняння виконує `runCheckSession` (`src/lib/checks/orchestrator.ts`,
блок `checks.has('ai')`): для кожного знайденого в БД ID бере снепшот з
`result.apiEntityMap` і викликає `compareEntityFields` — без жодного знання
про те, чи снепшот повний.

### Що хочемо: уніфікований механізм «повноти снепшота»

Механізм має бути **generic** — для будь-якого поточного чи майбутнього
адаптера з двоетапним збором, а не Ryde-специфічним хаком.

**Крок 1 — прапорець збагачення на сутності (адаптери).**
Адаптер позначає кожну сутність рівнем повноти снепшота:

```typescript
// у ScraperEntity (конвенція: службові ключі з префіксом _)
_snapshot?: 'detailed' | 'list_only'
```

- Ryde dockless: перші 50 на тайл → `'detailed'`, решта → `'list_only'`
  (поле відсутнє = `'detailed'` за замовчуванням — існуючі адаптери
  без двоетапного збору не змінюються).
- Мапінги читають лише свої `apiKey`, тож зайвий ключ нічого не ламає;
  у Manual Review side-by-side JSON він сам себе документує.
- Lyft pricings: капу підлягає сама ID-множина, прапорцем не вирішується —
  але той самий механізм дозволяє чесно показати часткове покриття
  (див. крок 3, `coverageNote`).

**Крок 2 — вердикт `Skipped` замість хибного `Different` (оркестратор).**
В `orchestrator.ts` перед `compareEntityFields`:

```typescript
if (apiSnapshot._snapshot === 'list_only') {
  await prisma.aiComparison.create({ data: {
    ..., verdict: 'Skipped',
    explanation: 'List-only snapshot (перевищено detail cap) — порівняння полів не виконувалось',
  }})
  continue
}
```

- Новий вердикт `'Skipped'` у типах (`AiVerdict`) + сірий бейдж в
  `AiResultsTab` / `ManualReviewPanel`.
- Повнота (`PolygonCheck` / `EntityCheckSummary`) не змінюється — ID
  залишаються в множині.

**Крок 3 — лічильники покриття в `EntityCheckSummary` (схема + UI).**

```prisma
model EntityCheckSummary {
  // ...
  detailedCount Int @default(0) // сутності з повним снепшотом
  listOnlyCount Int @default(0) // сутності понад detail cap
  coverageNote  String?         // напр. "pricings зібрані з 5 station_id з 2458 (Lyft cap)"
}
```

- UI: «1180 IMEI зібрано, 1102 збагачено деталями (93%)» + tooltip з
  поясненням капу; для Lyft pricings — `coverageNote` як warning-рядок.
- У 99% сесій буде 100% — лічильник насамперед сигналізує, *що кап взагалі
  спрацював*, і пояснює появу `Skipped`-рядків.

### Опційні продовження (поза мінімальним фіксом)

- фільтр «лише повні порівняння» у вкладці AI Results;
- конфігурований detail cap на формі створення сесії (для разових глибоких
  перевірок — безліміт, свідомо заплативши часом);
- для Lyft pricings — підняти кап або збирати станції не «перші 5»,
  а випадкові 5 (менша систематична сліпа зона).

### Критерій готовності

- list-only сутність ніколи не отримує вердикт `Different` через відсутні
  в снепшоті поля;
- у результатах сесії видно частку збагачених сутностей і причину пропусків;
- механізм задокументований у `docs/adding-new-scraper.md` як обов'язковий
  для адаптерів з двоетапним збором.
