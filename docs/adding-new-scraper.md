# Додавання нового скрапера

Цей документ описує кроки для інтеграції нового скрапера в систему data quality tool.
Кожен крок — це дослідження або рішення, яке треба прийняти перед написанням коду.

---

## 1. Визначити `appId`

Знайти `apps.name` скрапера в `scrapers_db.apps`. Це і є `appId` — рядковий ідентифікатор
(наприклад, `'ario'`, `'bird'`). Він використовується як ключ у `adapterRegistry` і
зберігається в `quality_db.Scraper.appId`.

> Не використовувати `apps.id` (integer) — він нестабільний між середовищами.

---

## 2. Дослідити API скрапера

- Ознайомитися з документацією або вихідним кодом скрапера (Python / JS / APK-розбір).
- Визначити базовий URL та схему автентифікації:
  - Bearer token? Session cookie? API key?
  - Де зберігаються облікові дані? (таблиця в `scrapers_db`, env-змінна, secrets manager)
  - Чи потрібне оновлення токена (refresh flow)? Як часто?
- Визначити необхідні заголовки HTTP (User-Agent, app version, custom headers).
- Перевірити, чи потрібні параметри пристрою (device ID, Android ID тощо).

---

## 3. Визначити ендпоінти для кожного типу сутностей

Для кожного з чотирьох типів (`dockless`, `docked`, `pricings`, `zones`):

- Чи підтримує скрапер цей тип взагалі? Якщо ні — `fetchEntities` повертає `[]`.
- Який HTTP-ендпоінт і метод (GET / POST)?
- Які параметри запиту (координати, місто, мова, ліміт тощо)?
- Де в JSON-відповіді знаходяться дані (шлях до масиву сутностей)?
- Яке поле є унікальним ідентифікатором сутності? (буде `id` в `ScraperEntity`)
  - Якщо нативного ID немає — потрібен детермінований UUID через `uuidv5`.

---

## 4. Визначити стратегію обходу полігонів

`polygonStrategy(entityType)` повертає `'all'` або `'center_only'`:

- **`'all'`** — запит відправляється для кожного полігону. Підходить для просторово
  залежних сутностей (транспортні засоби, станції доків).
- **`'center_only'`** — запит відправляється лише для центральної точки (або одного
  представницького полігону). Підходить для сутностей, що однакові по всьому місту
  (ціни, підписки, загальні зони).

Для кожного типу сутностей прийняти рішення на основі поведінки API:
- Чи повертає API різні дані для різних координат в межах одного міста?
- Чи достатньо одного запиту для отримання глобальних/міських даних?

---

## 5. Розібрати структуру відповіді та ID-маппінг

Для кожного типу сутностей:

- Задокументувати структуру JSON-відповіді (реальний приклад).
- Визначити поле `id` або спосіб його генерації через `uuidv5`.
- Вирішити, які поля залишати в `ScraperEntity`, а які відфільтрувати
  (наприклад, великі масиви координат краще виключати з diff-перегляду).
- Звернути увагу на відповіді, що сигналізують про блокування або rate-limiting:
  - `null` там, де очікується масив або об'єкт → кидати `ApiUnexpectedResponseError`.
  - HTTP 429, 503, або специфічні коди помилок скрапера.
- Переглянути Python-спайдер у `externalSystemDocs/` щоб зрозуміти точну логіку
  перетворень: назви полів, конвертації типів, обробку null-значень. Це критично
  для коректного `transform` у field mappings.

---

## 6. Дослідити схему scrapers_db для цього скрапера

Крім облікових даних, scrapers_db може містити додатковий контекст, необхідний для API-запитів.

**Облікові дані:**
- В якій таблиці зберігаються? (спільна `accounts` чи окрема для скрапера)
- Які поля потрібні для автентифікації?
- Написати helper-функцію в `src/lib/scrapers-db.ts` (тільки читання!).
- Якщо потрібен refresh — токен оновлюється тільки in-memory (не писати в `scrapers_db`).

**Таблиці сутностей:**
- Визначити таблиці, де зберігаються dockless, docked, pricings, zones для цього скрапера.
- Перевірити назви колонок — вони стануть `dbKey` у field mappings.
- Звернути увагу на типи даних: чи числа зберігаються як рядки? чи JSON-об'єкти серіалізовані
  як текст? (це впливає на стратегію `normalize` у маппінгу).

**Допоміжний контекст для API:**
- Деякі ендпоінти потребують параметрів, яких немає в `PolygonBounds` (наприклад, `location_id`,
  типи зон, міські ідентифікатори). Перевірити, чи є такі дані в scrapers_db.
- Якщо є — написати окремий helper для їх отримання (наприклад, `get<Scraper>ZoneContext`).

---

## 7. Визначити затримку між запитами

- `interPolygonDelayMs` — базова затримка між запитами до полігонів (за замовчуванням 500 мс).
- Перевірити, чи API скрапера чутливий до частоти запитів (rate limiting, тимчасові блокування).
- При необхідності збільшити базову затримку або додати випадковий jitter
  (jitter застосовується зовні, в логіці обходу полігонів).

---

## 8. Двоетапний збір і повнота снепшота (обов'язково для адаптерів з detail cap)

Деякі API віддають дані у **два етапи**: спершу список ID (легкий запит на регіон),
потім окремий detail-запит на кожну сутність для збагачення полів. Робити сотні
послідовних detail-запитів на полігон дорого, тож адаптер зазвичай **обмежує їх
капом** (напр. Ryde `MAX_VEHICLE_DETAILS = 50` на тайл). Це породжує **два
ортогональні класи неповноти**, які треба чесно позначати — інакше field-compare
дає хибні вердикти або сесія мовчки недозбирає дані.

### Клас A — неповний снепшот (`_snapshot`)

ID-множина **повна** (повнота «є в БД / немає» не страждає), але поля сутностей
понад кап доступні лише зі списку (`name = null` тощо). Якщо прогнати таку
сутність через field-compare, `name = null` проти реального значення в БД дасть
**хибний «Different»**.

Рішення — прапорець повноти на кожній сутності (`ScraperEntity._snapshot`):

```typescript
// збагачена сутність (у межах капу або з крос-ран кешу):
entity._snapshot = 'detailed'
// сутність понад кап (лише поля зі списку):
entity._snapshot = 'list_only'
```

- Відсутній прапорець = `'detailed'` за замовчуванням — **однокрокові адаптери
  нічого не змінюють**.
- Оркестратор сам робить решту: `'list_only'` → вердикт `Skipped` (без виклику
  field-compare) і лічильники `detailedCount` / `listOnlyCount` в
  `EntityCheckSummary`. Адаптеру достатньо проставити прапорець.

### Клас B — обрізана множина (`collectionNote`)

Кап ріже **саму ID-множину**: сутності поза капом взагалі відсутні в API-відповіді
(напр. Lyft збирає pricings лише з перших N станцій). Прапорцем це не вирішується —
сутностей нема на що його ставити. Натомість адаптер **декларує обмеження текстом**
через опційний метод:

```typescript
collectionNote(entityType: EntityType): string | null {
  if (entityType === 'pricings') {
    return `Pricings gathered from up to ${MAX} stations per polygon — coverage may be partial`
  }
  return null
}
```

Оркестратор пише цей рядок у `EntityCheckSummary.coverageNote` **дослівно**, не
знаючи про самі капи — UI показує його як warning. Кожен адаптер декларує власну
семантику.

### Опційні хуки життєвого циклу та конкурентність

Ці можливості теж **opt-in** (за замовчуванням поведінка не змінюється):

| Поле / метод | Призначення |
|---|---|
| `beginRun?(entityType)` / `endRun?(entityType)` | Викликаються навколо циклу полігонів одного типу. Використати для скидання per-run кешів — напр. крос-тайловий кеш деталей, коли тайли перетинаються і той самий ID збагачується повторно. |
| `maxConcurrentPolygons?` | Скільки полігонів обробляти паралельно (default `1` = послідовно). Пауза `interPolygonDelayMs` зберігається per-worker, тож глобальний RPS ≈ `workers / delay`. Вмикати **консервативно** й лише після смоуку — стежити за `suspectedBlock`. |

> **Критерій готовності для двоетапного адаптера:** list-only сутність ніколи не
> отримує `Different` через відсутні поля; у результатах видно частку збагачених
> сутностей і причину пропусків. Деталі механізму — `docs/known-issues.md` (issue #1).

## 9. Реалізувати та зареєструвати адаптер

1. Створити файл `src/lib/checks/adapters/<appId>-adapter.ts`.
2. Імплементувати інтерфейс `ScraperApiAdapter`.
3. Додати екземпляр до `adapterRegistry` в `src/lib/checks/adapters/scraper-adapter.ts`.
4. Переконатися, що `appId` в класі точно збігається з `apps.name` в `scrapers_db`.

**Важливо щодо констант:**
Якщо адаптер генерує значення детерміновано (не з API-відповіді, а з власної логіки) — ці
константи мають бути визначені і **експортовані з `field-mappings.ts`**, а не в адаптері.
Адаптер їх імпортує. Це забезпечує єдине джерело правди і коректне відображення в UI
(поле позначається як `constant`, а не як скопійоване з API).

## 10. Додати field mappings

Додати запис у `FIELD_MAPPINGS` в `src/lib/field-mappings.ts`. Це визначає, як система
порівнює API-снепшот з DB-записом у Field Check.

**Для кожного поля вирішити:**

| Параметр | Коли використовувати |
|----------|---------------------|
| `apiKey` + `dbKey` | Пряме зіставлення поля API з колонкою DB |
| `transform` | Перетворення значення з боку API перед порівнянням (наприклад, ділення на 100, маппінг enum) |
| `normalize` | Нормалізація **обох** сторін перед порівнянням (наприклад, JSON.parse для полів що зберігаються як рядок, вирівнювання глибини масивів координат) |
| `dynamic: true` | Поле змінюється між моментом збору API і запису в DB (GPS, заряд батареї) |
| `threshold` | Допустиме відхилення для `dynamic`-полів (дистанція в метрах, абсолютне або відсоткове значення) |
| `latPair` | Для GPS-пар: `dbKey` колонки з довготою, щоб розрахувати відстань через Haversine |
| `onlyWhen` | Поле релевантне тільки для певного підтипу сутності (предикат по API-снепшоту) |
| `constant` | Значення генерується адаптером, а не приходить з API; порівнюється напряму з DB |

**Підтипи сутностей (`onlyWhen`):**
Якщо один тип сутності має кілька підтипів з різними полями (наприклад, різні тарифні плани),
використовувати `onlyWhen`-предикати. Предикати визначати поряд з маппінгом у `field-mappings.ts`.

**`transform` vs `normalize`:**
- `transform` — застосовується тільки до API-значення (наприклад, конвертація типу або lookup).
- `normalize` — застосовується до **обох** сторін перед порівнянням. Використовувати, коли API
  і DB зберігають дані в різних форматах, що семантично еквівалентні (наприклад, `1` і `1.0`,
  або JSON-рядок і JSON-об'єкт, або різна глибина вкладеності масиву координат).

## 11. Написати unit-тести

Додати `describe`-блок у `src/lib/__tests__/field-compare.test.ts` для нового скрапера.

Мінімальний набір тестів для кожного типу сутностей:
- **Same** — еталонний кейс, де всі поля збігаються.
- **Different** — мінімум один кейс з розбіжністю для кожного статичного поля.
- **onlyWhen** — якщо є підтипи, перевірити що предикати коректно ізолюють поля.
- **normalize** — якщо є, перевірити що семантично рівні але структурно різні значення
  вважаються однаковими.

Для **двоетапних адаптерів** (секція 8) додати тест адаптера
(`src/lib/checks/adapters/__tests__/<appId>-adapter.test.ts`, приклад —
`ryde-adapter.test.ts`): сутності в межах капу → `_snapshot === 'detailed'`,
понад кап → `'list_only'`; `collectionNote` повертає ноту для потрібного типу
й `null` для решти.

Тести запускати через `npx vitest run` перед фінальною перевіркою в UI.

## 12. Перевірити в UI

1. Запустити `npm run scrapers-db:stage` або `npm run scrapers-db:prod` (port-forward).
2. Перейти на `/config` → **Sync from scrapers_db** — новий скрапер має з'явитися в списку.
3. Запустити сесію на `/sessions/new`:
   - Обрати новий скрапер.
   - Обрати кілька полігонів (починати з 2–3).
   - Запустити API→DB check для кожного типу сутностей окремо.
4. Перевірити результати на `/sessions/[id]`:
   - **Вкладка API→DB:** чи повертаються сутності? чи є помилки `ApiUnexpectedResponseError`?
   - **Вкладка Field Check:** відкрити кілька записів з verdict "Different" і переглянути
     DiffTable — чи відповідають правила (RULE колонка) очікуваним? Чи мають константні
     поля відмітку `constant`? Чи dynamic-поля не генерують хибних mismatch?
   - **Перевірити приклад "Same":** переконатися, що зелені рядки дійсно коректні, а не
     є хибнопозитивними через відсутній `normalize`.

## Орієнтовна структура файлу адаптера

```
src/lib/checks/adapters/
  scraper-adapter.ts    ← ScraperApiAdapter interface + adapterRegistry (редагувати)
  ario-adapter.ts       ← приклад повної реалізації
  mock-adapter.ts       ← мок для тестів
  <appId>-adapter.ts    ← новий адаптер

src/lib/
  field-mappings.ts     ← додати запис у FIELD_MAPPINGS + експортовані константи
  scrapers-db.ts        ← додати helper-функції для читання з scrapers_db

src/lib/__tests__/
  field-compare.test.ts ← додати describe-блок для нового скрапера

src/lib/checks/adapters/__tests__/
  <appId>-adapter.test.ts ← тест адаптера (для двоетапних: _snapshot, collectionNote)
```
