# Додавання нового скрапера

Цей документ описує кроки для інтеграції нового скрапера в систему data quality tool.
Кожен крок — це дослідження або рішення, яке треба прийняти перед написанням коду.

---

## 1. Визначити `appId`

Знайти `apps.name` скрапера в `scrapers_db.apps`. Це і є `appId` — рядковий ідентифікатор
(наприклад, `'ario'`, `'bird'`). Він використовується як ключ у `adapterRegistry` і
зберігається в `quality_db.Scraper.appId`.

> Не використовувати `apps.id` (integer) — він нестабільний між середовищами.

---

## 2. Дослідити API скрапера

- Ознайомитися з документацією або вихідним кодом скрапера (Python / JS / APK-розбір).
- Визначити базовий URL та схему автентифікації:
  - Bearer token? Session cookie? API key?
  - Де зберігаються облікові дані? (таблиця в `scrapers_db`, env-змінна, secrets manager)
  - Чи потрібне оновлення токена (refresh flow)? Як часто?
- Визначити необхідні заголовки HTTP (User-Agent, app version, custom headers).
- Перевірити, чи потрібні параметри пристрою (device ID, Android ID тощо).

---

## 3. Визначити ендпоінти для кожного типу сутностей

Для кожного з чотирьох типів (`dockless`, `docked`, `pricings`, `zones`):

- Чи підтримує скрапер цей тип взагалі? Якщо ні — `fetchEntities` повертає `[]`.
- Який HTTP-ендпоінт і метод (GET / POST)?
- Які параметри запиту (координати, місто, мова, ліміт тощо)?
- Де в JSON-відповіді знаходяться дані (шлях до масиву сутностей)?
- Яке поле є унікальним ідентифікатором сутності? (буде `id` в `ScraperEntity`)
  - Якщо нативного ID немає — потрібен детермінований UUID через `uuidv5`.

---

## 4. Визначити стратегію обходу полігонів

`polygonStrategy(entityType)` повертає `'all'` або `'center_only'`:

- **`'all'`** — запит відправляється для кожного полігону. Підходить для просторово
  залежних сутностей (транспортні засоби, станції доків).
- **`'center_only'`** — запит відправляється лише для центральної точки (або одного
  представницького полігону). Підходить для сутностей, що однакові по всьому місту
  (ціни, підписки, загальні зони).

Для кожного типу сутностей прийняти рішення на основі поведінки API:
- Чи повертає API різні дані для різних координат в межах одного міста?
- Чи достатньо одного запиту для отримання глобальних/міських даних?

---

## 5. Розібрати структуру відповіді та ID-маппінг

Для кожного типу сутностей:

- Задокументувати структуру JSON-відповіді (реальний приклад).
- Визначити поле `id` або спосіб його генерації через `uuidv5`.
- Вирішити, які поля залишати в `ScraperEntity`, а які відфільтрувати
  (наприклад, великі масиви координат краще виключати з diff-перегляду).
- Звернути увагу на відповіді, що сигналізують про блокування або rate-limiting:
  - `null` там, де очікується масив або об'єкт → кидати `ApiUnexpectedResponseError`.
  - HTTP 429, 503, або специфічні коди помилок скрапера.

---

## 6. Визначити облікові дані та їх зберігання

- Де зберігаються облікові дані для цього скрапера в `scrapers_db`?
  (таблиця accounts, окрема таблиця для кожного скрапера, або спільна)
- Які поля потрібні для автентифікації?
- Написати функцію-запит до `scrapers_db` (тільки читання!) для отримання активного облікового запису.
- Якщо потрібен refresh — токен оновлюється тільки in-memory (не писати в `scrapers_db`).

---

## 7. Визначити затримку між запитами

- `interPolygonDelayMs` — базова затримка між запитами до полігонів (за замовчуванням 500 мс).
- Перевірити, чи API скрапера чутливий до частоти запитів (rate limiting, тимчасові блокування).
- При необхідності збільшити базову затримку або додати випадковий jitter
  (jitter застосовується зовні, в логіці обходу полігонів).

---

## 8. Реєстрація в adapterRegistry

Після реалізації адаптера:

1. Створити файл `src/lib/checks/adapters/<appId>-adapter.ts`.
2. Імплементувати інтерфейс `ScraperApiAdapter`.
3. Додати екземпляр до `adapterRegistry` в `src/lib/checks/adapters/scraper-adapter.ts`:
   ```ts
   ['<appId>', new <AppId>ScraperApiAdapter()],
   ```
4. Переконатися, що `appId` в класі точно збігається з `apps.name` в `scrapers_db`.

---

## 9. Перевірити в UI

1. Запустити `npm run scrapers-db:stage` або `npm run scrapers-db:prod` (port-forward).
2. Перейти на `/config` → **Sync from scrapers_db** — новий скрапер має з'явитися в списку.
3. Запустити сесію на `/sessions/new`:
   - Обрати новий скрапер.
   - Обрати кілька полігонів (починати з 2–3).
   - Запустити API→DB check для кожного типу сутностей окремо.
4. Перевірити результати на `/sessions/[id]`:
   - Чи повертаються сутності?
   - Чи коректно відображається diff?
   - Чи є помилки `ApiUnexpectedResponseError` (ознака блокування)?

---

## Орієнтовна структура файлу адаптера

```
src/lib/checks/adapters/
  scraper-adapter.ts    ← ScraperApiAdapter interface + adapterRegistry (редагувати)
  ario-adapter.ts       ← приклад повної реалізації
  mock-adapter.ts       ← мок для тестів
  <appId>-adapter.ts    ← новий адаптер
```
